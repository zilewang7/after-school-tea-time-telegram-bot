# 技术设计：消息构建结构统一 + 系统提示词文件化

## 设计目标（用户视角）

模型看到的每条消息应该是一个**格式统一、机器可读**的结构，而不是现在这样的括号汤——`(forwarded from user ඞ丁二 τ2 Bjutähndeiohl 🐈🥬🐛) (I send a picture)` 这种正文为空时两个风格各异的英文括号连排。统一后模型更容易理解元信息（转发来源/回复对象/引用/附件），回复质量与稳定性更好。同时系统提示词从 .env 的单行字符串（`\n` 转义地狱）改为多行 markdown 文件维护，改提示词不再需要在一行里数转义。

## 现状问题

存库文本与构建期拼接混用了四种风格的元信息标注：

| 元信息 | 现状写法 | 产生位置 |
|---|---|---|
| 媒体附件 | `(I send a picture)` 第一人称英文括号 | autoSave 存库文本 |
| 转发来源 | `(forwarded from user X)` 英文括号 | autoSave 存库文本 |
| 回复上下文 | `([system]replying to [名字:摘要][quote: 引文]): ` | context-builder 构建期 |
| 结束标记 | `<<EOF\n` | autoSave 存库文本 |
| 系统通知 | `[system] …` 独立文本段 | 多处 |

结构性问题：**元信息被烧进存库文本**，导致 ① 三处代码靠正则从文本里抠回来（`chat-handler.ts:117` 抠媒体名、`retry-handler.ts:84/181` 剥 `(I send …)` 提取 pic 提示词、媒体下载失败后 autoSave 整段重写 text）；② 想改展示格式必须动存库；③ 正文为空的消息（纯转发图）产出上面那种括号汤。

## 方案

### A. 元信息出文本、进列（messages 表 +2 列）

- `mediaHint: string | null` — 存 CapturedMedia 的 hint（`a picture` / `a video…`），下载失败时 autoSave 异步更新这一列（追加 `, too large to process` 等状态），**不再重写 text**。
- `forwardOrigin: string | null` — 存 `user 张三` / `channel 某频道` / `chat 某群`。
- `text` 回归**纯正文**（markdown），不再追加 `(I send …)`、`(forwarded from …)`、`<<EOF`。
- sqlite + sequelize sync alter 自动加列，无迁移脚本；`sub image of [id]` 内部标记本轮不动（它不进模型上下文，仅内部查询用）。

### B. 构建期统一渲染格式

context-builder 把所有元信息渲染成**用户名与冒号之间的方括号标注段**（与现有 replying to 位置一致），`<<EOF` 移到构建期追加：

```
用户名 [forwarded from user 某某] [replying to 某某: "开头摘要"] [quote: "引文"] [sent a picture]: 消息正文
<<EOF
```

- 标注全部 `[key …]` 方括号风格；`[system]` 前缀只保留给真正的系统通知（媒体感知 nudge、媒体失败说明等独立段落）。
- 媒体失败态渲染进标注：`[sent a picture — failed to download, you cannot see it]`。
- **媒体 nudge 按能力与类型收敛（用户补充）**：现在的 AUDIO_VISUAL_NUDGE 是 text part，媒体过滤器只删 media part，导致仅图片能力的模型也收到"你能感知音视频"的提示。改为：① nudge 只在 `capabilities.supportsMediaInput` 为真且消息确实带该类媒体时插入；② 按实际类型措辞——视频只说 video、音频只说 audio，不再笼统"audio/visual"；③ 模型能力不支持该媒体（被过滤掉）时，标注渲染为 `[sent a video — not visible to you]`，让模型明确知道自己看不到，而不是沉默丢弃。
- 正文为空时格式依然成立：`用户名 [forwarded from user X] [sent a picture]: ` — 不再是括号汤。
- history 消息与 current 消息用同一个渲染函数（现在是两套相似代码，顺带合并）。

### C. 三处正则依赖改读列

- `chat-handler.ts:117`（等待下载时抠媒体名）→ 直接读 `mediaHint` 列。
- `retry-handler.ts` picbanana/picgpt 提示词提取 → 不再需要剥 `(I send …)`（text 已是纯正文）；**保留旧正则一周**兼容存量行（7 天自动清理），并注释届时可删。
- autoSave 失败态更新 → 只 update `mediaHint` 列。

### D. 系统提示词文件化 + 重写

- 新增宿主机文件 `system-prompt.md`（与 .env 同级，**不入 git**，加入 .gitignore），多行 markdown 自由维护。
- `.env` 中 `SYSTEM_PROMPT=<单行>` 替换为 `SYSTEM_PROMPT_FILE=system-prompt.md`；`getSystemPrompt()` 改为：有 `SYSTEM_PROMPT_FILE` 则读文件（启动时读一次缓存），否则回退 `SYSTEM_PROMPT` 环境变量（兼容不便挂文件的场景）。
- `docker-compose.yaml` 给主/测试容器加挂载 `./system-prompt.md:/app/k-on-bot/system-prompt.md:ro`；`.env.test` 同步改。
- 不选 dotenv 多行引号方案的原因：compose `env_file:` 与 dotenv 双通道解析同一个 .env，compose 注入的（可能截断的）环境变量优先于 dotenv，多行值在两个解析器间行为不一致，是个坑。

### E. 新系统提示词草稿（`system-prompt.md`，随格式一起审）

```markdown
# 角色
你是 Telegram 群聊 bot「K-ON」(@AfterSchoolTeatimeBot)，正在回复 Telegram 群聊中的消息。
你喜欢使用 emoji（不使用颜文字）。除非用户另有要求，始终用中文回复（哪怕用户名不是中文）。

# 你收到的消息格式
每条用户消息形如：

    用户名 [标注…]: 消息正文
    <<EOF

- 用户名和冒号之间可能有零或多个方括号标注：
  - `[forwarded from user/chat/channel 某某]` — 这是转发的消息及其来源
  - `[replying to 某某: "开头摘要"]` — 用户回复了哪条消息
  - `[quote: "引文"]` — 回复时引用的具体片段
  - `[sent a picture]`（或 video/sticker/audio…）— 消息附带的媒体；
    标注里出现 failed / too large 时说明该媒体你看不到
- 正文为空 = 用户只发了媒体，或只是召唤你没留内容
- `<<EOF` 是消息结束标记
- 独立的 `[system] …` 段落是系统信息，不是用户说的话
- 最后一条消息是发给你的，回复它；你的回复直接输出正文，**不要**模仿上述格式

# 能力与输出
- 输出支持完整 Markdown（标准语法+部分方言）：防剧透 `||spoiler||`、折叠块
  `<details>/<summary>`（summary 可选）、引用 `>`、任务列表 `- [x]` 等
- 多使用搜索工具联网查证；不要把任何问题预设为简单问题

# 注意
- 用户可能编辑自己已发出的消息，或切换你生成的消息的版本
```

（语义与旧提示词一致，仅格式章节替换为新结构 + markdown 化排版。）

## 验证

- 扩展 `scripts/e2e/rich-inject.mts` 或新增注入断言：转发+媒体消息落库后 text 为纯正文、`mediaHint`/`forwardOrigin` 列正确。
- 手动构建一次 context（临时脚本或单测）目检渲染串：转发纯图消息应为 `名字 [forwarded from user X] [sent a picture]: `。
- e2e 全量 `pnpm test:e2e:bot`（含 picbanana 提示词提取回归——如无该用例则手测一次 /picbanana retry）。
- 上线后群里实测：转发消息、带图消息、回复+引用，问 bot 能否正确理解来源。

## 风险

- 存量 7 天窗口内的旧行 text 仍带 `(I send …)<<EOF`，构建期会再包一层新格式 → 元信息重复但语义不破坏，7 天自愈；不做存量清洗。
- `getRepliesHistory`/版本系统若有对 `<<EOF` 的隐式依赖需排查（grep 目前只见 retry-handler 剥除处）。
- compose 挂载新文件：文件不存在时容器起不来（bind mount 目录化坑）——上线顺序：先建文件再改 compose。
