# 测试 bot 自动化端到端测试(luoxu userbot 驱动)

## 设计目标(用户视角)

改完 k-on-bot 后,不再需要你亲自去 Telegram 里戳测试 bot:AI 助手(Claude)能自己启动测试容器、以真人账号在测试群发消息触发 @WatchFirstBot、读回它的真实回复(文本+entities+编辑过程)并断言,最后干净下线——全程不打扰主群:测试 bot 启动先丢弃积压事件、只响应测试群、下线时清掉命令列表和简介,主群输入 `/` 永远不会看到测试 bot 的命令。

## 现状(已核实)

- **触发条件**(src/util.ts:44 `checkIfMentioned`):@提及 / 回复 bot 消息 / 私聊。**没有任何群组白名单** → 测试 bot 在主群被 @ 或被回复就会响应,这就是"无故打扰"的来源
- **启动**(src/index.ts:87):`bot.start()` 裸调,无 `drop_pending_updates` → 停机期间主群积压的 @ 会在启动瞬间被批量处理
- **命令/简介**:`cmd/loader.ts:12` `setMyCommands` 默认 scope(全局可见);index.ts:74-81 无条件 set 简介 → 测试 bot 每次启动都把自己注册成"看起来像正式 bot",且**下线后命令仍留在服务端**(当前就处于这个状态,需一次性清理)
- **停机**:无 SIGTERM 处理(测试容器 `node dist/index.js` 直跑,docker stop 即杀)
- **驱动通道**:bot 收不到其他 bot 的消息(Telegram 限制),必须真人账号驱动。luoxu(~/dockers/luoxu-bot)的 Telethon userbot 已登录、**已在测试群 2311128579**(在 index_groups 里);luoxu-api(aiohttp,容器名固定,traefik-net 与 k-on-bot 互通)已有自定义 `/preview` 端点先例(k-on-bot 的 luoxu-preview-service 在用)——**在 src-api 加端点是本部署的成熟做法**
- **验证通道**:①(主)测试 bot 自己的 sqlite(`database.test.sqlite`,挂载在本地工作树,**宿主机可直接读,已实测**)——`bot_responses.versions` 存完整回复文本,按 `userMessageId` 关联、`buttonState != 'processing'` 即终态,轮询它就是最可靠的完成信号;②(辅,验证"群里实际可见内容/entities")MTProto 读回,走 luoxu 新端点
- **测试容器**:compose profile `test`,本地工作树挂载 + 独立 DB + .env.test,已可用;走云端 Bot API(经 SOCKS),不依赖 local-bot-api;前台 `node dist/index.js` 直跑,SIGTERM 可直达进程
- **注意**:@WatchFirstBot 的 token 同时被独立项目 ~/dockers/watch-first-bot 使用(当前未运行)——启动测试容器前需确认它没在跑,否则 getUpdates 抢占冲突

## 方案

### 1. 测试实例行为隔离(k-on-bot 代码,均由 env 驱动,生产不受影响)

`.env.test` 新增:

```
TEST_INSTANCE=1
ALLOWED_CHAT_IDS=-1002311128579   # 逗号分隔;未设置 = 不限制(生产现状)
```

- **群组门控**:chat-handler 入口 + cmd 各 handler 统一前置检查——`ALLOWED_CHAT_IDS` 存在且当前 chat 不在列表 → 直接 return(不回复、不入库)。实现放 `checkIfMentioned` 同层的独立函数,一处接入 `registerChatHandler`(bot.on 最外层)+ cmdLoad 包装
- **启动丢弃积压**:`TEST_INSTANCE=1` 时 `bot.start({ drop_pending_updates: true })`(生产保持现状,不丢消息)
- **命令 scope 收窄**:`TEST_INSTANCE=1` 时 `setMyCommands(..., { scope: { type: 'chat', chat_id: <ALLOWED第一项> } })`,且**不**设置全局简介(跳过 setMyDescription/ShortDescription)→ 主群 `/` 永远不出现测试 bot 命令(比"下线时删"更根本,崩溃也不泄漏)
- **优雅停机**:注册 SIGTERM/SIGINT handler:`deleteMyCommands()`(default scope + chat scope,双保险)→ `bot.stop()` → exit。生产同样受益(目前 docker stop 是硬杀)
- **一次性清理**:实施时手动跑一次 `deleteMyCommands` + 清简介,把测试 bot 当前遗留在服务端的全局命令/简介清掉

### 2. luoxu-api 测试驱动端点(src-api,仿 /preview 模式)

```
POST /luoxu/test/send      {g, text, reply_to?}   → {message_id}   # userbot 发消息
GET  /luoxu/test/messages  ?g=&min_id=&limit=     → [{id, sender_id, text, entities, edit_date, reply_to}]
```

- 复用现有 Telethon client(与 /preview 同一实例,无并发会话问题)
- **安全门**:两个端点仅接受 `g=测试群`(服务端硬编码白名单,拒绝主群)——杜绝"内网任意程序借 userbot 在主群发言/读史"
- entities 原样透传(Telethon → JSON),供断言 telegram-md-entities 渲染结果

### 3. 测试编排脚本(k-on-bot 仓库 `scripts/e2e/`)

`pnpm test:e2e:bot` 流程:
1. 确认 ~/dockers/watch-first-bot 未运行(token 冲突检查),`docker compose --profile test up -d k-on-bot-test`,等日志出现 "Bot started"
2. 经 luoxu `/test/send` 发 `@WatchFirstBot <用例>` 到测试群
3. 主验证:轮询 `database.test.sqlite` 的 bot_responses(按 userMessageId,`buttonState != 'processing'` 即终态,超时 90s);辅验证:`/test/messages` 读回群内实际文本+entities
4. 断言:回复文本/entities、DB versions 一致性
5. teardown:`docker compose stop k-on-bot-test`(SIGTERM 触发命令清理)

首批用例:普通对话回一条、markdown 重格式消息(验证 entities 渲染)、流式长回复分条、429 部分内容保留(配合文档一)、/model 命令。

Claude 日常使用方式:直接跑步骤脚本,或单独 curl 两个端点做临时验证——**全程无需你出场**。

## 风险与边界

- userbot 发言频率:测试用例串行 + 每条间隔 ≥3s,避免账号 flood 风险;e2e 不进 CI,只本机手动/Claude 触发
- luoxu-api 重启会短暂中断主 bot 的链接预览(可接受,秒级)
- 测试群消息会被 luoxu 正常索引(无妨,本就是测试群)
- bot-to-bot 不可见的限制意味着**无法**绕过 userbot;若某天 luoxu 下线,备选是 gramjs 独立 session(需你扫码/验证码登一次)

## 实施顺序

1. k-on-bot:门控 + drop_pending + 命令 scope + SIGTERM(纯增量,先上测试容器自验)
2. 一次性清理测试 bot 遗留命令/简介
3. luoxu-api:两个端点 + 白名单,重启 api 容器
4. 编排脚本 + 首批 5 用例
5. 跑通后把用法写进 memory(供跨会话使用)
