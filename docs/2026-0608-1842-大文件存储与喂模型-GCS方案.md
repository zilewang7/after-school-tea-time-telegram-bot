# 大文件存储与喂模型：GCS 引用式方案（2026-0608-1842）

> 承接《[2026-0608-1645] Local Bot API 突破 20MB 下载限制》。那份解决**下载层**（拿到 >20MB 字节）；本文档补齐**存储层 + 喂模型层**，三层串成端到端。

## 1. 设计目标（从用户视角）

用户 @bot 发送或引用大视频 / 大文件（≤200MB）时，bot 不仅能**下载**，还能真正**看懂并回答**——大文件以"引用式"喂给 Gemini，而**不把字节灌进 SQLite**（已 922MB）、不在内存反复 base64、不被模型 ~20MB 的 inline 上限卡住。大文件**不在本地长期留存**：经 GCS 中转喂给模型，bucket **1 天自动删**（用户已确认可接受过期失效——超 1 天的历史大文件降级为文本提示）。整套**计费走小号 $300 赠金**（token 大头），资源建在大号项目统一管理。

## 2. 现状链路：处处内联 base64（大文件的根本障碍）

```
下载/读盘 → Buffer → SQLite BLOB(MediaCache.data) → toString('base64')
        → UnifiedContentPart.{imageData|mediaData}
        → Gemini inlineData{ data: base64 }  /  OpenAI data:...base64
```

- 每一环全字节复制：SQLite 膨胀、内存 1.33×、请求体超 Gemini inline **~20MB** 上限。
- `media`（音视频）**实际只有 Gemini 消费**（`message-transformer.ts:61-78` 的 `inlineData`）；OpenAI/Grok 只吃 image。

## 3. 已确认的关键事实（探针 + context7）

1. **能吃大媒体的只有 Gemini**，且必须"引用式" `fileData.fileUri`，inline 受 ~20MB 限制。
2. **Vertex 用不了免费 Files API**（`files.upload` 在 Vertex 抛错）；Vertex 的大文件引用走 **GCS `gs://`**。
3. **探针实测**（容器内 express 配置）：`fileData.fileUri` 引用**公开** `gs://` 成功（`GS_OK: Scones`）。私有 bucket 需读取方身份被授权——故采用 **SA 同项目**方案（SA 天然能读同项目私有 bucket）。
4. SDK `@google/genai@2.7.0`；当前 `new GoogleGenAI({ vertexai:true, apiKey })`（express）；请求 `generateContentStream({ model, contents })`。

## 4. 总体方案：三层

| 层 | 解决 | 方案 |
|---|---|---|
| 下载层 | 拿到 >20MB 字节 | local bot api `--local`（1645 文档），读盘 |
| **存储层** | 不灌 SQLite | 大文件**流式上传私有 GCS**（lifecycle 1 天删）；MediaCache 存 `gs://` 引用而非 BLOB；小文件维持现状 |
| **喂模型层** | 喂 Gemini 大文件 | Gemini `fileData.fileUri`（`gs://`）；小文件 inline；非 Gemini 平台降级文本 |

**按大小分流**（阈值 `INLINE_MAX_BYTES`）：
- **≤阈值**：维持现状——存 BLOB、inline base64。零 GCS 往返/成本。
- **>阈值**：上传 GCS → 存 `gs://` → Gemini `fileData.fileUri`。

> **阈值 = 20MB，与 TG getFile / 现状 `MAX_MEDIA_BYTES` 对齐**。≤20MB 维持现状 inline（实测 Vertex 能吃：18.4MB 原始 / ≈24.5MB base64 正常工作），>20MB（local bot api 新解锁的部分）才走 GCS。
>
> 走 GCS 的理由**不是** "inline 撞限"——Vertex inline 不卡 20MB（其设计是小文件 inline、大文件 GCS），Developer API 那个 20MB 也已于 2026-01 升到 100MB。真正理由是**避免大文件 base64 灌进 SQLite BLOB（库已 922MB）+ 撑大请求体/内存**。这样 inline 段一行不改，只在 >20MB 新增 GCS 路径，改动最小、语义统一（三个 20MB：TG 下载上限 / 现状 MAX_MEDIA_BYTES / 本阈值，天然一致）。

## 6. 代码改动（方向性）

### 6.1 新增 GCS 上传服务
- 依赖：`@google-cloud/storage`（用 ADC / SA JSON）。
- `src/services/gcs-service.ts`：
  ```ts
  import { Storage } from '@google-cloud/storage';
  const storage = new Storage(); // ADC
  const bucketName = process.env.GCS_BUCKET;

  // 流式上传 local bot api 落盘文件，不 readFile 全进内存
  export const uploadToGcs = async (localPath: string, objectName: string, mime: string): Promise<string> => {
      await storage.bucket(bucketName).upload(localPath, { destination: objectName, contentType: mime });
      return `gs://${bucketName}/${objectName}`;
  };
  ```
- 对象名用 `fileUniqueId`（content-addressed，去重）+ 扩展名。

### 6.2 `UnifiedContentPart` 增加引用字段（`types.ts`）
```ts
export interface UnifiedContentPart {
    type: 'text' | 'image' | 'media';
    text?: string;
    imageData?: string;
    mediaData?: string;    // 仅小文件 inline 时有值
    fileUri?: string;      // gs:// 引用（大文件）
    mimeType?: string;
    mediaKind?: string;
}
```

### 6.3 `message-transformer.ts`：media 分支按有无 `fileUri` 分流
```ts
.with({ type: 'media' }, (p) => {
    const mimeType = p.mimeType ?? 'application/octet-stream';
    const mediaPart: GeminiPart = p.fileUri
        ? { fileData: { fileUri: p.fileUri, mimeType } }   // 大文件：gs:// 引用
        : { inlineData: { mimeType, data: p.mediaData ?? '' } }; // 小文件：inline
    // sticker fps 采样逻辑不变
    return mediaPart;
})
```
- `GeminiPart` 增加 `fileData?: { fileUri: string; mimeType: string }`。
- **非 Gemini 平台**（openai/grok/...）遇到 `fileUri` 的 media part：降级为文本提示（如 `[用户发送了一个视频，当前模型无法查看]`），它们本就只吃 image。

### 6.4 媒体保存流程（`autoSave.ts`）
- 现状：下载 → `putCachedMedia`（BLOB）。
- 改造：拿到字节/盘文件后按大小分流——
  - ≤阈值：维持 `putCachedMedia`（BLOB）。
  - >阈值：`uploadToGcs(盘文件路径)` → `putCachedMedia` 只存 `fileUri`（不存 `data`）。
- 配合 local bot api：大文件本就落盘，**直接流式上传盘文件**，上传后删盘（见 1645 §4.6）。

### 6.5 `MediaCache` / `context-queries.ts`
- `mediaCacheDTO`：增加 `fileUri`（nullable）；`data` 对大文件 nullable；保留 `createdAt`。
- `getFileContentsOfMessage`：cached 有 `fileUri` → 产出 `{ type:'media', fileUri, mimeType }`；否则 BLOB → 现状 inline。
- **过期失效（1 天，对齐 autoClear）**：现有 `autoClear` 已有"媒体字节超 1 天清空 `Message.file`"逻辑（`autoSave.ts:501`），本方案与之并列：
  - MediaCache 的 `gs://` 引用同样 **超 1 天即失效**，由 `autoClear` 一并清理（删 MediaCache 行 + 删对应 GCS 对象）。
  - `getFileContentsOfMessage` 对超 1 天的引用产出文本提示 `[媒体已过期，超过 1 天]`，不再引用 `gs://`。
  - bucket lifecycle 设 **2 天**作兜底删除（> 判定的 1 天），避免"GCS 已删但仍被引用"的边界 404。

## 7. 各模型 / 各场景行为

| 场景 | 行为 |
|---|---|
| ≤20MB 图/媒体 + Gemini | 现状 inline（不走 GCS）|
| >20MB 视频 + Gemini（24h 内）| GCS `gs://` 引用，正常理解 |
| >20MB 视频 + Gemini（超 1 天历史）| `gs://` 已删 → 降级文本提示 |
| 任意大媒体 + OpenAI/Grok/Claude | 降级文本提示（本就不支持）|

## 9. 风险与缓解

| 项 | 风险 | 缓解 |
|---|---|---|
| free trial 跨账号 billing | — | ✅ 已实测可关联 |
| SA 凭据 | JSON 泄露 | `:ro` 挂载、不进 git、最小权限 |
| inline 大小 | 大文件 base64 撑 SQLite/请求体 | 阈值 20MB（=现状/TG 边界），>20MB 走 GCS（见 §4）|
| `gs://` 边界 404 | lifecycle 删与判定竞态 | 判定 1 天（autoClear 主清）、lifecycle 2 天兜底 |
| 非 Gemini 平台 | 大文件无法处理 | 降级文本提示 |
| 改动面大 | 多处（types/transformer/autoSave/cache/queries）| 不分步（目标完整才可验证）；探针 + 本地测试逐环节验证后一次上线 |
