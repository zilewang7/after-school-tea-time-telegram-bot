# 技术设计：富文本消息媒体保存 + luoxu 富文本落库 + 预览图 OCR 增强

## 设计目标（用户视角）

群友用会员富文本编辑器发的消息，两套系统都要"完整可用"：① k-on-bot 要能看到消息里的图片——现在问 bot"他发了什么"，bot 只能复述文字，图片根本没存下来；② luoxu 搜索要能命中富文本消息——现在这类消息落库 text 为空，`/s` 永远搜不到；③ 顺带增强：消息里链接的网页预览图也做 OCR 进索引，预览图上的文字（截图、海报、推文配图）变得可搜。

> 诊断样本：主群 msg 1065858（表格 + 图片 + 任务列表的富文本消息）。
> k-on-bot 落库：markdown 正常（`[图片] dagbjz` 占位），但 `file`/`fileUniqueId` 为 null。
> luoxu 落库：整行 text 为空字符串。

## 根因

**k-on-bot**（`src/db/autoSave.ts`）：`resolveCapturedMedia` 只检查 `msg.photo / sticker / voice / audio / video / video_note / document`，从不看 `msg.rich_message.blocks`。而 Bot API 的 `RichBlockPhoto.photo` 就是标准 `PhotoSize[]`（含 file_id，可正常 getFile 下载），video/audio/voice_note/animation 块同理——媒体信息全在，只是没人去取。

**luoxu**（`src-api/luoxu/indexing.py`）：`_format_msg` 只读 `msg.message / poll / web_preview / media`。富文本消息在 MTProto 层 `message=''`、`media=None`，内容在 `msg.rich_message`。容器内 telethon 1.44.0（TL layer 227）**已原生支持**该字段：

```
RichMessage { blocks: List[TypePageBlock], photos: List[Photo], documents: List[Document], rtl, part }
```

blocks 复用 Instant View 的 PageBlock 类型族；photos/documents 是完整媒体对象（可直接 `client.download_media`）。luoxu 什么都没读 → 空 text 落库。所有入库路径（历史回灌 + NewMessage + MessageEdited）都汇聚到 `format_msg`，**只需改一处**。

## 方案

### A. k-on-bot：提取富文本消息里的图片

1. 新增 `resolveRichMessageMedia(richMessage)`: 递归遍历 blocks（photo/video/animation/audio/voice_note 是叶子；blockquote/details/collage/slideshow/list item 会嵌套 blocks），收集媒体块。
2. **v1 决策：只取第一个图片类媒体**（photo 块优先；没有 photo 时取 video/animation）。理由：
   - 现表结构一条消息一个 file 槽位（`fileMime`/`fileUniqueId` 单列），多媒体要动 schema + buildContext，改动量不成比例；
   - 富文本最多 50 媒体，但实际场景几乎都是一图。
   - 多于一个媒体时 hint 写明：`(I send a rich message with N images, only the first is attached)`。
3. 下载/缓存/GCS 全复用现有 `acquireMediaBytes`——photo 块给出的就是普通 file_id。
4. `autoSave` 中 `resolveCapturedMedia(msg) ?? resolveRichMessageMedia(msg.rich_message)`；`autoUpdate`（编辑富文本消息）同样补上。

### B. luoxu：富文本消息落库 + 图片 OCR

1. `indexing.py` 新增 PageBlock → 纯文本 flattener：
   - 文本类块（paragraph/header/subheader/title/list/ordered list/table/blockquote/pullquote/pre/details/footer…）逐类取 RichText 文字（RichText 本身也是递归结构，写一个 `richtext_to_str`，telethon 有现成的各 RichText 类型）；
   - 媒体类块输出 `[image]`/`[video]`/`[file]` 占位；
   - 未识别块类型兜底跳过（记 debug 日志），保证永不抛错。
2. `_format_msg` 增加分支：`if rm := getattr(msg, 'rich_message', None):` → flatten blocks 进 text；遍历 `rm.photos` 逐张 OCR（本地服务，不设张数上限）。
3. `ocr.py` 小改：`ocr_img` 目前只接受 `MessageMediaPhoto`/带 document 的 media；放宽为也接受裸 `Photo`/`Document`（cache key 取 `.id`，`download_media` 原生支持裸对象），mime 按对象类型判断。
4. 存量空行回填：写一次性脚本（仿 `ocr_backfill.py`），扫 `text=''` 的行按 msgid 重拉消息重新 format + update。1065858/1065865 顺带修复。

### C. 增强：网页预览图 OCR + Instant View 内容入库

- 现状：`web_preview` 的 url/site_name/title/description 文字**已在索引**；缺预览图 OCR 和 Instant View 正文。
- 预览图：`_format_msg` 的 webpage 分支里，`w.photo` 存在时对其 OCR，输出 `[webpage image]` + OCR 行。复用 B-3 放宽后的 `ocr_img`，photo.id 缓存天然去重（同一链接刷屏不重复 OCR）。
- **Instant View 正文（零成本，直接入库）**：`WebPage.cached_page` 是 `Page { blocks: PageBlock[], photos, documents }`——与 `rich_message` 同一套 PageBlock 结构，内容**随消息自带，无需任何额外抓取**。存在时用 B-1 的同一个 flattener 展开进 text，前缀 `[instant view]`。
  - 长度上限：IV 文章可能整篇很长，flatten 后截断到 4000 字符（config 可调），够搜索命中即可；
  - 图片：`cached_page.photos` 同样全部过 OCR（本地服务，不设上限）；
  - `part=True`（服务端只给了部分页面）时照常处理，有多少收多少。
- 不做：document 型预览（视频封面）。

## 验证

- k-on-bot：扩展 `scripts/e2e/rich-inject.mts` 注入带 photo 块的 update，断言 walker 选中第一张图、hint 正确（file_id 是假的，下载失败路径本来就有 `failed to download` hint 兜底，正好一起验）。真机渲染验证靠你用会员号往测试群发一条带图富文本。
- luoxu：flattener 用 1065858 的真实结构做离线单测（telethon 对象手工构造）；上线后测试群发富文本 → `/s` 关键词命中 + OCR 文字命中。
- 回填脚本跑主群，确认 1065858/1065865 行有内容。

## 风险

- PageBlock 类型族较大（Instant View 遗产），flattener 漏类型 → 兜底跳过不抛错，漏的只是少收录不是崩。
- luoxu 是 lilydjwg/luoxu 的 fork，改动收敛在 `indexing.py` + `ocr.py` 两文件，便于日后跟上游。
- OCR 量增加（预览图 + 富文本图）：paddle 本机服务 + 3600s photo.id 缓存，量级可控；如有压力可对 webpage 图加开关（config `ocr_webpage_preview`）。
