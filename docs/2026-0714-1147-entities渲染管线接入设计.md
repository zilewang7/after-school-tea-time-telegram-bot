# telegram-md-entities 接入 k-on-bot 技术设计

> 2026-07-14 · 前置文档:`~/projects/telegram-md-entities/docs/2026-0713-1021-telegram-md-entities-技术设计文档.md`(§8 为本方案概要版)、`docs/2026-0710-1452-流式编辑429优化与终态一致性.md`(edit coordinator 现状)

## 1. 设计目标(最终用户视角)

在 Telegram 里和 bot 对话的用户,AI 回复的格式**永远正确渲染**:不再出现 `\*\*` 转义残渣、不再有"格式解析失败整条消息退化成转义纯文本"、代码块恢复语法高亮(语言标记不再丢失)、中文表格渲染为清晰的层级列表、长回复跨消息拆分时粗体/引用等格式无缝续接、流式过程中半截的粗体/代码块也按最终样式实时显示而不是闪烁的字面符号。对开发者(自己):显示层从"转义+解析+降级链"的猜测式管线变成 `{text, entities}` 的精确数据管线,长度就是 `text.length`,拆分是纯数据操作,全部离线可测。

## 2. 前置:包 review 结论(2026-07-14,已完成)

对 `telegram-md-entities@0.1.0` 的全量 review 结论:**可发布**。

- 质量:typecheck 干净、89/89 离线测试全过、build/publint/attw 全过、无 TS 规范违例、无 TODO/死角;e2e 17 语料真实回环 0 失败(对照 telegramify-markdown 管线 3 个硬 400)。
- 已修复(commit `c7e7cfb`):缺 LICENSE 文件;`dist/browser.js`(~1.08MB,仅 playground 用)误入 tarball——排除后包体 1.3MB → **46.7kB**;`displayWidth` 死代码删除。
- API 面(`renderMarkdown` / `splitMessage` / `validateMessage` / `wrapInBlockquote` / `concatMessages` / `toPreviewHtml`)覆盖 bot 接入全部需求,无缺口(流式 tick 全量重渲 <1ms,无需增量 API)。

## 3. 总体方案

**渲染层换轨**:所有 LLM 输出的展示路径改为 `renderMarkdown(raw)` → `{text, entities}`,发送/编辑时传 `entities`,**不传 parse_mode**。DB 始终存 raw markdown(现状如此),零数据迁移——版本切换、refreshToFinalState、retry 自动受益。

```
                    ┌ 流式:renderMarkdown(buffer, {streaming:true})
raw buffer (DB) ────┤
                    └ 终态:renderMarkdown(buffer) → splitMessage(4096/90)
         thinking ── renderMarkdown → wrapInBlockquote(流式=blockquote, 终态=expandable)
         组装 ────── concatMessages(thinking, '\n', text, grounding, stats)
         发送 ────── edit-coordinator: editMessageText(text, { entities })   ← 无 parse_mode
```

### 3.1 决策点 A:包如何进 Docker(已拍板 2026-07-14)

**先不发布 npm**(等真实使用验证后再发)。开发/验证期用 `link:../../projects/telegram-md-entities` 本地依赖 + 测试容器挂载包目录(见 3.3);验证通过、准备上线时再发布 npm 并把依赖切成 registry 版本(线上 Dockerfile 是 fresh clone + pnpm install,本地路径不可解析,发布是上线硬前置)。

### 3.2 决策点 B:灰度策略(已拍板 2026-07-14)

**直接切换,不保留旧管线**,旧代码(转义/降级链/可见长度估算)随迁移一次性删除,顺带优化现有代码质量。理由:

1. 双轨意味着 ~10 个文件、两套类型(`parseMode` vs `entities`)并存,灰度期间旧路径无人验证,是纯负担;
2. 包侧已用真实 Bot API 做过 17 语料回环 + 拆分回环 + 实体上限探测,风险已在包内消化;
3. 全程在测试容器验证(见 3.3),线上部署前已实测;回滚也只需 git revert + docker restart。

保留的安全网(比旧降级链简单一个量级):

- **coordinator 兜底**:`editMessageText`/`sendMessage` 携带 entities 返回 400 时,剥掉 entities 以纯文本重发(内容必达,格式让位)并 `console.error` 留痕。理论上不应触发(`validateMessage` 离线保证合法),属死人开关。
- **静态 UI 不动**:`cmd/menu.ts`、`help.ts` 等手写 Markdown 的非 LLM 路径保留 `parse_mode`(coordinator 同时支持两个字段,互斥使用),不在本次范围。

### 3.3 决策点 C:测试容器(已拍板 2026-07-14)

用测试 bot `@WatchFirstBot` 起独立容器 `k-on-bot-test`(compose profile `test`),开发全程不影响线上:

- 挂载**本地工作树**(非 git clone)→ 改代码即可重启验证,无需 push;
- 挂载 `~/projects/telegram-md-entities` 到容器内匹配 `link:` 相对路径的位置;
- 独立 `node_modules` named volume(alpine musl 与宿主机 glibc 原生模块隔离)、独立 sqlite DB(`DB_PATH` 环境变量,顺带把 `db/config.ts` 的硬编码路径改为可配)、独立 `.env.test`(测试 bot 凭据,其余密钥复用线上 `.env`);
- 复用 warp 代理与 traefik-net 网络。

线上容器与流程完全不动;验证通过后才 push master + 重启线上容器。

## 4. 改造点(按依赖顺序分四个 Stage)

### Stage 0:本地依赖接线 + 测试容器

k-on-bot `package.json` 加 `"telegram-md-entities": "link:../../projects/telegram-md-entities"`;新增 compose `k-on-bot-test` 服务、`.env.test`、`DB_PATH` 支持。先用**未迁移的现有代码**把测试容器跑通(隔离容器基建风险与代码迁移风险)。上线前:npm publish + 依赖切 registry 版本。

### Stage 1:发送层支持 entities(向后兼容,可独立部署)

- `src/telegram/edit-coordinator.ts`
  - `DesiredState` / `SubmitEditOptions` 增加 `entities?: MessageEntity[]`(与 `parseMode` 互斥);`flushEntry` 传 `{ entities }`(edit-coordinator.ts:381 处)。
  - **去重修正**:`matchesSentState`(edit-coordinator.ts:219)目前只比 `sentText` + markup,文本相同仅实体变化会被误判"已发送"——增加 `sentEntitiesJson` 对比。
  - `handleEditError` 的 `can't parse entities` 分支(:293)改为上文的"剥 entities 纯文本重发"兜底;`buildFallbackText` 管道保留至 Stage 4 删除。
- `src/telegram/streaming-editor.ts`:`updateContent` 透传 entities;状态行仍是尾部追加纯文本(append 不影响前面实体的 offset,无需调整);`getStatusTextEscaped` → 纯文本。
- `src/telegram/message-editor.ts`:`EditOptions` 增加 entities。

### Stage 2:终态路径切换

- `src/telegram/formatters/final-message-builder.ts`:`buildFinalMessageChunks(): string[]` → `buildFinalMessages(): RenderedMessage[]`:
  1. thinking:`wrapInBlockquote(renderMarkdown(thinking), true)`(expandable);
  2. text:`renderMarkdown(text)`;
  3. grounding / agent-stats(`grounding-formatter.ts`、`agent-stats-formatter.ts`):改为**生成 markdown 源**(`**Sources**`、`[title](url)`、`|` 分隔),render 后 `wrapInBlockquote(…, true)`,视觉与现状一致,手工转义全删;
  4. `concatMessages` 组装 → `splitMessage`(4096 字符 + 90 实体双预算;跨界实体自动闭合重开,thinking 跨消息的 self-closing 语义由包保证,替代 `splitThinkingIntoCollapsedChunks`)。
- `src/reply/response-handler.ts` `sendFinalResponse`(:488):消费 `RenderedMessage[]`,首条走 `editor.updateContent(…, {isFinal})`,其余 `sendMessage(text, { entities, reply_parameters })`;删 parse-error retry(:546)。
- `src/services/bot-message-service.ts`:`switchVersion`(:542)、`refreshToFinalState`(:810)改用新 builder;`createSession` 占位符 `'✽ Thinking\\.\\.\\.'` → 纯文本 `'✽ Thinking...'`;`handleSessionError` 手工转义(:433)→ 纯文本。

### Stage 3:流式路径切换

- `src/reply/response-handler.ts` `formatStateForDisplay`(:198)→ 返回 `RenderedMessage`:
  - thinking:`wrapInBlockquote(renderMarkdown(thinkingBuffer, {streaming:true}), !isProcessing)`(流式普通引用,终态 expandable,与现状语义一致);
  - text:`renderMarkdown(textBuffer, {streaming: isProcessing})`;
  - `concatMessages` 拼接。
- **continuation 拆分双预算**(:355-456):触发条件从"可见长度 ≥ 3900−50"改为 `rendered.text.length ≥ 3850 || rendered.entities.length ≥ 85`;raw 切点复用现有 `splitRawByFormattedLength` 骨架,但 measure 从 `getTelegramVisibleLength(escape(…))` 换成 `renderMarkdown(rawPrefix).text.length`(**精确**,render <1ms,二分次数 ~12 次无性能问题);段落/换行 tier-first 边界策略不变。
- 所有预转义字面量(`'continued'`、`\[stopped\]`、`\[Empty response\]` 等)→ 纯文本。

### Stage 4:清理(随迁移一次性完成,见决策点 B)

删除:`escapeMarkdownV2` 及全部调用、`toTelegramMarkdown`、`formatResponse`/`formatResponseSafe`/`formatThinkingContent`、`getTelegramVisibleLength` + MarkdownV2 unit helpers、`smartSplit`/`needsSplit`、streaming-editor `buildFallbackText`/`setRawParts`/`RawMessageParts`、coordinator `buildFallbackText` 管道、bot-message-service `editWithFallback`、依赖 `telegramify-markdown`。`truncateForTelegram`(纯文本错误消息用)保留。

## 5. 不变的部分

- **DB 与上下文构建**:存取均为 raw markdown,零改动。
- **按钮/回复参数/链接预览**:`reply_markup`、`reply_parameters`、`link_preview_options` 与 entities 正交,原样透传。
- **coordinator 的预算/429/final-sticky 模型**:完全复用,仅增加 entities 字段。
- **入站侧** `autoSave.ts` 的 entity 处理(读用户消息):不属于本次范围。

## 6. 风险与对策

| 风险 | 对策 |
|---|---|
| Docker fresh clone 装不到包(**顶级风险**) | Stage 0 先发布;部署前在容器内验证 `pnpm install` |
| 实体变化文本不变被去重吞掉 | Stage 1 的 `sentEntitiesJson` 对比(有单测) |
| 流式大 buffer 实体数先于长度超限 | continuation 双预算触发(≥85 实体) |
| 异常 entities 触发 400 | coordinator 剥 entities 纯文本兜底 + 留痕 |
| thinking 跨消息折叠语义 | 包 `splitMessage` 跨界闭合重开已实测;bot 侧再实测一次长 thinking |
| 状态行追加破坏 offset | 追加在尾部,不影响已有实体,无需处理(已推演) |

## 7. 验证方案

1. **单测**(bot 仓库):`formatStateForDisplay` / `buildFinalMessages` 输出快照(text+entities);coordinator entities 去重与 400 兜底分支。
2. **测试容器实测**(`k-on-bot-test` + `@WatchFirstBot`,在测试群):流式长回复(观察格式实时渲染、continuation 拆分点、第一条不缩水)、含代码块/表格/thinking/grounding 的完整回复、版本切换、停止按钮、对旧消息 refreshToFinalState。
3. **上线**:npm publish + 依赖切 registry → push master → 重启线上容器;观察日志无 `BAD_REQUEST`、无 fallback 触发。
