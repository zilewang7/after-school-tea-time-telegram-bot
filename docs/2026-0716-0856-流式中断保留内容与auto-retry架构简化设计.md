# 流式中断保留部分内容 + @grammyjs/auto-retry 架构简化 + 依赖升级

## 设计目标(用户视角)

当模型流式输出到一半报错(典型:Gemini 429 RESOURCE_EXHAUSTED)时,用户已经看到的大段内容**不再被整条替换成"请求过于频繁，请稍后重试"**——内容原样保留,错误信息以一行不显眼的斜体追加在末尾,并保留重试按钮;切换版本、重试后回看,行为一致。同时把散落在各处的手写 Telegram API 重试兜底交给官方 auto-retry 插件,代码更轻、行为更稳;顺带把落后的依赖升到位。

## 一、流式中断保留部分内容

### 现状与根因(已核实)

错误显示有**两条重叠路径**,后者覆盖前者:

1. `handleSessionError`(src/services/bot-message-service.ts:428)——其实已经做对了:
   - DB:`finalizeSession({errorMessage})`,错误存 `version.errorMessage` **独立字段**;`version.text` 和 Message 表只存真实部分内容 → **DB 侧现状已满足"报错不进消息内容"**
   - 显示:thinking 引用块 + 部分正文 + 斜体 `Error: ...` 追加
2. `handleResponseError`(src/reply/response-handler.ts:796)——所有调用方(chat-handler / retry-handler)走的入口:
   - 先 `await session.handleError(error)`(上面那条,显示正确)
   - 然后**再次编辑**,内容 = `formatErrorForUser(error)` 纯错误文案——`partialText` 参数从未有调用方传过,`chatContext.currentState` 里攥着的已流出内容被白白丢弃
   - 最终可见状态 = 纯错误文案 ← **用户看到的 bug 就在这**

另一个一致性缺口:版本切换重建显示走 `buildFinalMessages`(bot-message-service.ts:582),不带 `version.errorMessage` → 切到出错版本时错误提示凭空消失;`refreshToFinalState`(:819,Stop 按钮自纠错触发)同理会剥掉错误行。

**续写消息的加重情形**:长回复续写后 `session.editor`(第一条消息)与 `chatContext.editor`(最后一条续写)指向**不同消息**——当前错误会不一致地出现在两条消息上。修复后错误行归属**最后一条消息**(chatContext.editor 所有者)。

**调用面**:`handleResponseError` 共 19 个调用点;有部分内容的只有 6 个 processStream 失败点(chat/retry/picgpt/picbanana),其余 12 个是流前错误(此时 buffer 为空,自动退化为纯错误文案)——从 `chatContext.session` 内部取 buffer 即可统一覆盖,**无需改任何调用方**。

### 方案

1. **统一错误显示构建器**(新增,放 `src/telegram/formatters/`):
   `buildErrorDisplay({ text, thinking, errorMessage })` → thinking(折叠引用)+ 正文(strict 渲染)+ `\n\n` + 斜体错误行(truncate 500)。空内容时退化为纯错误文案(现行为)。
2. `handleResponseError` 重构:
   - 部分内容取 `chatContext.currentState`(processStream 每 chunk 同步),兜底 `session.textBuffer/thinkingBuffer`
   - 用统一构建器渲染,一次编辑到位;按钮逻辑(HAS_VERSIONS/RETRY_ONLY)保留现有实现
   - `session.handleError` 退化为**只做 DB finalize**,删除其中的显示编辑(消除双编辑竞态)
3. **版本切换/刷新重建**:`buildFinalMessages` 调用处(切版本 bot-message-service.ts:582、`refreshToFinalState` :819)在 `version.errorMessage` 存在时经统一构建器**重新追加**错误行(否则用户点一次 Stop 自纠错后错误提示会消失)——错误行永远是**显示期合成**,不进任何存储
4. `formatErrorForUser` 的文案映射保留(429→"请求过于频繁"等),作为错误行的内容来源

### 不变量

- DB(`version.text` / Message.text / modelParts)永不含错误文案
- 有部分内容 ⇒ 部分内容全部可见,错误行只追加不替换
- 重试按钮行为不变;重试成功后新版本正常,旧错误版本切回时仍显示"内容+错误行"

## 二、@grammyjs/auto-retry 评估与接入

### 结论:采纳,但只替换"单调用重试",不动 edit-coordinator 的核心价值

auto-retry 是 grammy 官方 API transformer:任何 API 调用遇 429 flood-wait 按 `retry_after` 自动等待重放,可配 `maxRetryAttempts` / `maxDelaySeconds`,超限后才抛错。

现状盘点(手写兜底,src/telegram/edit-coordinator.ts):
- **可删**:`runApiCall` 的手动 429 重试循环(:574-600)——auto-retry 全覆盖
- **必须保留**:edit-coordinator 的**合并语义**(流式编辑 latest-state-wins、每聊天预算与节流)——auto-retry 只会把过时的中间态编辑排队重放,恰恰是有害的;这块是自研价值所在
- **简化**:coordinator 内部对 429 的显式退避(:196-211, :284-307)可以收窄——接入 auto-retry 后 429 基本不再穿透到业务层,pacing 预算负责"少发",auto-retry 负责"万一还是 429 就兜住"

### 接入方式

```ts
bot.api.config.use(autoRetry({ maxRetryAttempts: 3, maxDelaySeconds: 60 }));
```

- `runApiCall` 保留"每聊天串行 + 预算"外壳,删除内部重试循环
- edit-coordinator 保留 pacing/coalescing/sticky delivery,429 处理分支简化为日志(理论上到不了)
- "message is not modified" 等非 429 分支不变(auto-retry 不管这些)
- 净效果:删掉一段手写重试状态机,新增 1 个依赖 + 1 行接线

## 三、依赖升级(pnpm outdated 实测)

分两批:

**批 1(低风险,一次性升)**:@google-cloud/storage 7.21、@google/genai 2.12(429 行为无 breaking,changelog 过一遍)、grammy 1.44(auto-retry 需要较新版)、openai 6.47、tsx 4.23、@sequelize/core+sqlite3 alpha.48(同步升,alpha 系列锁 minor)

**批 2(major,逐个评估,本轮默认不动,除非你点头)**:
- typescript 5.5→7.0:大版本,等生态稳;先升到 5.9 与包仓库对齐
- dotenv 17 / socks-proxy-agent 10 / sqlite3 6:breaking 面小但要跑通测试容器验证
- @types/node 26:**不升**——容器还是 node:18;若你愿意把 Dockerfile/测试容器一起升到 node:22 LTS(推荐,18 已 EOL),则 types 跟着到 22

验证:typecheck + build + 测试容器跑一轮真实对话(依托文档二的 e2e 能力更佳)。
