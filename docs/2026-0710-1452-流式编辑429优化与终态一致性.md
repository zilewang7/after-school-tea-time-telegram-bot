# 流式编辑 429 优化与终态一致性设计

## 设计目标（用户视角）

在群聊里同时 @ 机器人发起多个对话时，每条回复都能平稳地流式刷新，不再频繁撞 Telegram 的 429 限流；无论中途发生什么（限流、网络抖动、用户点停止），消息最终一定会刷新到完整、正确的最终文本，不会卡在半截的中间态；当消息卡住时点"停止"按钮，会把消息内容刷新到当前最新的正确状态，而不是只把按钮删掉；长回复拆分到下一条消息时，第一条消息只会变多不会变少，且拆分点落在段落/换行处，不会从句子中间截断。

## 一、现状与根因分析

### 问题 1：多并发回复必撞 429

涉及文件：`src/telegram/rate-limiter.ts`、`src/telegram/streaming-editor.ts`

1. **限流器只"等待"不"占位"（惊群）**。`waitForRateLimit` 读共享的 per-chat 计数算出 delay 后 sleep，`recordEdit` 只在 API 成功后才写回。N 个并发 editor 同时读到相同状态 → 算出相同 delay → 同时醒来同时发请求，突发流量直接触发 Telegram flood 限制。
2. **无 429 感知**。`doEdit` 失败只 `console.error` 后返回 false。GrammyError 携带 `parameters.retry_after`（日志里的 "retry after 21"），代码没有读取：不退避、不重试，下一个定时编辑立刻再撞，形成 429 风暴。
3. **本地预算与远端不同步**。失败的请求不 `recordEdit`，但 Telegram 端 429 的请求同样计入 flood 计数，导致本地越是"以为还有配额"越撞。

### 问题 2：卡中间态后点停止只删按钮

涉及文件：`src/reply/response-handler.ts`、`src/cmd/menus/response-menu.ts`

卡中间态的成因链：流结束 → `sendFinalResponse` 里 `session.finalize()` 先把 DB 的 `buttonState` 更新为终态 → 随后的 final `editMessageText` 撞 429 → `doEdit` 返回 false **且不重试** → 消息文本永久停留在中间态，而 DB 已认为完成。

此时用户点停止：`handleStop` 发现 `buttonState !== PROCESSING`，走 `refreshButtonsToCurrent`——它只调 `editMessageReplyMarkup` 刷按钮（终态多为 NONE → 按钮被移除），**不刷新消息文本**。这正是"点了停止只是按钮没了"的现象。

### 问题 3：拆分导致第一条消息"缩水"、句中截断

涉及文件：`src/reply/response-handler.ts`（流式拆分）、`src/telegram/formatters/smart-splitter.ts`

1. **拆分点用系数估算，严重保守**。流式拆分时对 raw 文本调 `smartSplit(text, Math.floor(availableSpace / 1.5))`（thinking 用 `/1.2`），系数是对"转义后长度膨胀"的拍脑袋估计。中文文本转义后几乎不膨胀，实际能放 ~3800 可见字符的第一条被截到 ~2500 → 客户端上已经显示出来的内容被"收回"挪到下一条，第一条肉眼可见地变短。
2. **边界查找对中文不友好**。`findSplitPos` 把换行和空格同等对待，中文段内没有空格、窗口内可能没有换行时直接落到 `lastSafeBoundary` 硬切句中。且它把 raw 未转义文本按 MarkdownV2 规则解析（`*_~`>` 等被当格式符跳过），对 raw 文本会误判可见长度。

### 问题 4：冗余编辑是 429 的最大流量源

涉及文件：`src/telegram/streaming-editor.ts`

1. **纯装饰 spinner 编辑**。每个活跃 editor 空闲时每 2.5~5s 发一次编辑，仅为了把 `◐ Processing...` 换成 `◑ Analyzing...`。n 个并发回复 = n × 12~24 次编辑/分钟，而整个群共享约 20 次/分钟的预算——两个并发就必然超支，spinner 编辑和内容编辑互相争抢配额，双输。
2. **编辑不合并**。每次 `updateContent` 都是独立的一次 API 调用诉求，即使排队等待期间已经有更新的内容到来，旧内容依然可能被发出（版本检查只在等待结束后判一次），中间版本白白消耗配额。
3. **刷新频率不感知并发**。`updateInterval = 500ms`、`idleInterval = 2500ms` 都是常量，不随同 chat 活跃回复数缩放。

## 二、总体设计：per-chat Edit Coordinator（期望状态模型）

核心思想：把"我要调一次 editMessageText"改为"我提交这条消息的**最新期望状态**"。每个 chat 一个协调器（coordinator），统一持有该 chat 的编辑预算，按节拍逐条把"期望状态 ≠ 已发送状态"的消息刷到最新。

```
StreamingEditor / MessageEditor / sendFinalResponse
        │  submit(messageId, desiredState, priority)
        ▼
┌────────────────────────── per-chat EditCoordinator ──────────────────────────┐
│ desiredMap: messageId → { text, markup, parseMode, priority, isFinal }       │
│ sentMap:    messageId → 已成功发送的 text/markup                              │
│ drain loop: 按节拍取一条 dirty 消息（final > content > spinner）→ 调 API      │
│ 429: 读 parameters.retry_after → 整 chat 暂停到 penaltyUntil，状态不丢         │
└──────────────────────────────────────────────────────────────────────────────┘
```

这一个模型同时解决问题 1 与问题 4：

- **天然合并（coalescing）**：drain 时永远取该消息**当前最新**的期望状态，排队期间到达的中间版本从不占用 API 调用。"skip if same content" 也变成结构化的 `desired === sent` 判断。
- **天然串行 + 占位**：一个 chat 只有一个 drain loop，不存在惊群；节拍间隔即预算，`interval = max(minEditInterval, windowMs / maxEditsPerMinute)`，并发消息多时每条消息的实际刷新频率自动摊薄，无需显式调参。
- **优先级**：`final > content > spinner`。预算紧张时 spinner 自动饿死（不发也无所谓），content 优先；final 最优先且 sticky（见下）。
- **429 全局退避**：任何一次调用返回 429，读 `retry_after`，整个 chat 的 drain 暂停到 `penaltyUntil = now + (retry_after + 1)s`。期间期望状态照常更新，恢复后直接发最新版——**限流不再导致卡中间态，只是刷新变慢**。
- **sendMessage 同预算**：continuation / final chunks / 图片发送通过 coordinator 的 `acquireSendSlot()` 走同一预算与同一 penalty，替代现在裸调的 `waitForRateLimit + recordEdit`。

### 终态一致性（问题 2）

1. **final 状态 sticky**：`isFinal` 的期望状态提交后，coordinator 会持续重试（遵守 retry_after，指数退避，上限约 5 次/2 分钟）直到成功；成功前该消息一直算 dirty。这从根源上消灭"DB 完成但客户端停在中间态"。
2. **停止按钮语义修正**：`handleStop` 的非 PROCESSING 分支改为调用新的 `refreshToFinalState(chatId, firstMessageId)`——从 DB 取当前版本，`buildFinalMessageChunks` 重建终态文本，`editMessageText`（文本+按钮）刷新，而非只刷 reply markup。chunks 与该版本 `messageIds` 逐条对应编辑；数量对不上时至少把点击的那条与 `currentMessageId` 刷成对应 chunk。该函数同时作为 final 重试次数耗尽后的人工兜底入口。
3. PROCESSING 分支保持现状（abort → 正常流程收尾），正常流程收尾本身已被 1 保证送达。

### 精确拆分（问题 3）

替换系数估算为**精确测量**，新增 `splitRawByFormattedLength(raw, format, maxVisible)`：

1. 在 raw 文本上做二分/递进查找，候选前缀经真实 `format`（即流式路径实际使用的 `escapeMarkdownV2` / thinking 引用格式化）后用 `getTelegramVisibleLength` 测量，找出格式化后 ≤ maxVisible 的最大前缀。拆分低频（每 ~4000 字一次），性能无虞。
2. 边界优先级：在 `[maxVisible × 0.75, maxVisible]` 窗口内依次找 `\n\n`（段落）> `\n` > 句末标点（`。！？.!?`）> 空格；窗口内都没有才硬切。窗口下限保证不会为了找边界丢弃过多内容。
3. **不缩水保证**：拆分在显示长度达到 `limit - 50` 时才触发，而精确拆分的第一条保留到 ~limit，必然 ≥ 已显示内容，第一条只增不减。
4. `findSplitPos` 现有的"raw 文本按 markdown 解析"问题随之消失：raw 路径用新函数；已格式化文本路径（`final-message-builder`）继续用现有 `smartSplit`，仅同步加入相同的边界优先级。

## 三、实施计划（4 个 PR 粒度的阶段）

| 阶段 | 内容 | 改动文件 | 效果 |
|------|------|----------|------|
| P1 | 新增 `edit-coordinator.ts`（drain loop、优先级、429 penalty、send slot），`rate-limiter.ts` 退役为其内部实现 | telegram/ | 消灭惊群与 429 风暴 |
| P2 | `StreamingEditor`/`MessageEditor` 改为提交期望状态；spinner 降为最低优先级；删除独立 idle 编辑竞争 | streaming-editor.ts、message-editor.ts、response-handler.ts | 冗余编辑归零，多并发自动摊薄 |
| P3 | final sticky 重试；`refreshToFinalState`；`handleStop` 非 PROCESSING 分支接入 | bot-message-service.ts、response-menu.ts | 终态必达；停止按钮符合预期 |
| P4 | `splitRawByFormattedLength` + 边界优先级；替换 `/1.5`、`/1.2` 估算 | smart-splitter.ts、response-handler.ts | 拆分不缩水、按换行拆 |

P1/P2 耦合较紧建议一起上；P3、P4 相互独立可并行。

## 四、风险与兼容

- **行为变化**：spinner 在多并发时基本不再转动（被 content 编辑饿死），单回复时不受影响——这是刻意取舍，装饰性更新让位于内容送达。
- **coordinator 生命周期**：per-chat 单例挂在现有 app state；消息 final 送达后从 map 清除，chat 无 dirty 消息时 drain loop 停摆，无内存增长。
- **重启丢状态**：coordinator 状态在内存中，与现状一致（现有 rate limiter 也在内存），不新增风险。
- **retry_after 解析**：优先读 `GrammyError.parameters.retry_after`，缺失时回退正则解析 message，再缺省 30s。

## 五、验证方案

1. 单元：coordinator 的合并/优先级/penalty 逻辑；`splitRawByFormattedLength` 对中文、含 markdown、长 thinking 的拆分点断言（≥ 已显示长度、落在换行/句末）。
2. 实测：同一群同时发起 3~4 个 mention，观察日志无 `429`、每条回复持续刷新且都到达终态；流式中途手动制造长回复触发拆分，确认第一条不缩水、拆在换行处；用 iptables/断网手段让 final 编辑失败，确认重试送达，以及点停止能全文刷新。
