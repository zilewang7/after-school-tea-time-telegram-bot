# 双触发与媒体组去重加固（2026-0605-1832）

## 1. 设计目标（从用户视角）

用户在群里给 bot 发消息时，无论是**单条** GIF / 视频 / 文件，还是**一组**多图 / 多视频 / 多文件（相册），bot 都只回复一次：
- 不出现重复的「✽ Thinking…」占位消息；
- 不对同一个问题生成两条独立回复；
- 后台不再因重复处理刷 `bot_responses.messageId` 唯一约束错误。

即便 Telegram 因网络抖动或 bot 重启而**重投**同一条 update，bot 也能识别并跳过，不重复回复。

## 2. 根因回顾（已排查确认）

通过还原出事 chat（`-1001502100157`，12149~12154）的数据库现场 + 进程/日志核验，确认：

1. **不是 media_group 引起**：出事的三组消息间隔 5~6 分钟手动重发、是真实的「回复自己的 GIF 再 @bot」关系、无 `media_group_id`。
2. **不是多实例**：当前仅 docker 容器一个实例独占 sqlite，日志无 `409 Conflict`、无 UNIQUE 报错。
3. **直接原因**：旧版 `createBotResponse` 用裸 `BotResponse.create`，**非幂等**——同一逻辑回复被处理两次即撞 UNIQUE。已由 `upsert`（commit `0afe8bf`）兜底，报错消除，但「同一消息生成两次回复 / 两条占位」的隐患仍在。
4. **被放大的真实隐患**：commit `0afe8bf` 把监听 filter 从 `['msg:text','msg:photo','msg:sticker']` 扩展到含 `'msg:video'`/`'msg:document'` 等。而 `shouldSkipMessage` 的相册去重**只判 `ctx.message?.photo`**，导致 video / document 类型的相册去重彻底失效，相册每个成员各回一次。

> 这两条隐患正交：第 3 条是「同一条消息（相同 message_id）被重复处理」，第 4 条是「相册多条不同消息（相同 media_group_id）只该回一次」。需要分别治理。

## 3. 改动方案

### 3.1 修复相册去重（`src/reply/chat-handler.ts`）

把 `shouldSkipMessage` 的判断由「是图片」改为「属于已记录的某个媒体组、且不是该组首条」，覆盖所有媒体类型：

```ts
const shouldSkipMessage = (ctx: Context): boolean => {
    const mediaGroupId = ctx.update?.message?.media_group_id;
    if (!mediaGroupId) return false;

    const mediaGroupTemp = getMediaGroupIdTemp();
    return (
        mediaGroupTemp.chatId === ctx.chat?.id &&
        mediaGroupTemp.mediaGroupId === mediaGroupId &&
        mediaGroupTemp.messageId !== ctx.message?.message_id
    );
};
```

要点：
- `mediaGroupTemp` 由 `autoSave` 中间件（运行在 `handleReply` 之前）按「组内首条」写入，时序成立；
- 首条消息 `messageId === temp.messageId`，不跳过、正常回复；后续成员被跳过；
- 去掉 `ctx.message?.photo` 限制，video / document / 混合相册一并生效。

### 3.2 入口幂等去重（`src/reply/chat-handler.ts` + `src/state.ts`）

在 `handleReply` 决定要回复后、创建会话前，按 `chatId:userMessageId` 做幂等：同一条用户消息若**正在处理或刚处理过**，直接跳过。

`state.ts` 仿照现有 `editMonitorMap`（带过期 + 容量上限清理）新增一个注册表：

```ts
// state.ts —— 已处理/处理中的用户消息，key = "chatId:userMessageId"
const PROCESSED_TTL_MS = 5 * 60 * 1000; // 覆盖 Telegram 重投窗口
const MAX_PROCESSED = 200;

export const tryMarkUserMessageHandling = (chatId: number, userMessageId: number): boolean => {
    // 返回 true 表示「抢到处理权」，false 表示「已被处理/处理中，应跳过」
    // 内部做过期清理与容量上限淘汰（同 addEditMonitorEntry 的模式）
};
```

`handleReply` 入口：

```ts
if (!checkIfMentioned(ctx, options?.mention)) return;
if (shouldSkipMessage(ctx)) return;

// 幂等：抢不到处理权说明这条消息已在处理/已处理，直接跳过
if (!tryMarkUserMessageHandling(ctx.chat.id, ctx.message.message_id)) return;
```

要点：
- 标记**保留** `PROCESSED_TTL_MS` 窗口而非处理完即删——这样 Telegram 在该窗口内重投同一 update 时仍能被识别跳过；
- 仅作用于 `handleReply` 普通对话主路径；`retry` / 版本切换走 callback，不经此入口，不受影响；
- 进程重启会清空注册表（内存态），重启瞬间的重投仍依赖 `upsert` 兜底——属于可接受的残余概率。

> 备选（更强、但改动更大）：在 `createSession` 非 retry 分支按 `(chatId, userMessageId)` 查 `bot_responses`（已有该复合索引），命中则跳过。优点是重启也幂等，缺点是触碰会话创建核心路径、且有查-插竞争窗。**本次倾向先用内存注册表方案**，`upsert` 仍作为最终防线保留。

## 4. 不做的事

- **不清理进程**：已确认无 k-on-bot 残留（`1661842`/`1662547` 属其它项目，sqlite 仅容器独占）。
- **不移除 `upsert`**：作为最后一层防御保留。
- **不改 `setTimeout` 脱钩结构**：本次只在其入口加幂等，不重构中间件链（避免牵动 picbanana/picgpt/chat 的现有时序）。

## 5. 影响范围与风险

| 改动 | 影响面 | 风险 | 缓解 |
|---|---|---|---|
| 3.1 相册去重 | `shouldSkipMessage` | 若 `mediaGroupTemp` 未及时写入，首条可能被误判 | 时序上 `autoSave` 先于 `handleReply`，已成立；首条用 `messageId` 区分 |
| 3.2 入口幂等 | `handleReply` 入口 + `state` | TTL 内同一 message_id 的「合法二次主动触发」会被挡 | 同一 message_id 不会被用户二次主动触发；retry 走 callback 不受影响 |

## 6. 验证方式

1. 单条 GIF + @bot：只回一次，无重复占位，日志无 UNIQUE。
2. 多视频/多文件相册 + @bot：只回一次（修复前会多回）。
3. 图片相册：保持只回一次（不回归）。
4. 人为重复投递（或快速重启）同一条消息：第二次被幂等跳过。
