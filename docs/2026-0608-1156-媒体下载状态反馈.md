# 媒体下载状态反馈（2026-0608-1156）

## 1. 设计目标（从用户视角）

用户 @bot（或回复 bot、用 `/chat`、`/picbanana`、`/picgpt`）时，如果**本次对话上下文里有图片/视频/文件还在后台下载**，bot 会**立刻**回一条「⏳ 正在获取媒体…」让用户知道在等什么，而不是几分钟毫无动静。媒体就绪后这条提示**无缝消失**、换成正常的「✽ Thinking…」占位继续回答；若某个媒体**下载超时或失败**，提示**不消失而是就地改成失败说明**（哪个媒体、为什么失败、它不会进入对话上下文，并附报错），用户一眼知道当前状态。

## 2. 现状与关键问题

延迟处理（`chat-handler.ts` 的 `setTimeout`）当前顺序：

```
waitForFileSave([当前消息, reply目标])   // ← 在最前面、静默
  → checkPic* → handlePic*(发 processing) → 返回
  → dealChatCommand   // /chat：把「被回复消息+后续N条」写进 replies，改变上下文树
  → handleReply → buildContext → AI
```

**问题（这次修订的核心）**：`buildContext` 真正读媒体的范围是 `getRepliesHistory` 遍历出的**整条 reply 树**（沿 `replyToId` 找根 + 递归 `replies`）。而 `/chat` 的 `dealChatCommand` 会往 `replies` 里塞消息**改变这棵树**，它却跑在 `waitForFileSave` **之后**——所以在最前面等待时，上下文树还没定型，等待范围（只猜 `[当前, reply]`）必然漏掉 `/chat` 引入的媒体，也漏掉 reply 链深层的媒体。

**结论**：等待必须**下沉到上下文树定型之后、`buildContext` 之前**，且范围取自 `getRepliesHistory`。

成功/失败信号（`autoSave`）：媒体处理结束后 msgId 从 `asyncFileSaveMsgIdList` 移除；**成功**写入 `fileUniqueId`，**失败**在消息文本留 `[system] too large/failed to download/failed to render` 标记且无 `fileUniqueId`。

## 3. 方案

### 3.1 先判「会不会回复」+ 幂等前移

- 新增无副作用预判 `willReply(ctx)`：文本以 `/(picbanana|picgpt|chat)` 开头，或 `checkIfMentioned`（@bot / 回复 bot / 私聊）。不满足直接退出——不等待、不提示，群里无关消息不受影响。
- 把 `tryMarkUserMessageHandling` 从 `handleReply` 入口**前移到 `setTimeout` 顶部**（`willReply` 之后），并从 `handleReply` 移除。防止重复触发发出两条提示，并让 pic 路径也获得幂等保护。

### 3.2 等待时机：按路径放在「上下文确定后」

```
setTimeout:
  if !willReply(ctx) return
  if !tryMarkUserMessageHandling(...) return
  text = message.text || caption
  if text 是 /picbanana|/picgpt:
      await awaitMediaWithFeedback(ctx, [当前, reply])   // 参考图就绪后再 check 读图
      checkPic* → handlePic* → return
  else:
      await dealChatCommand(ctx)        // ← 先让 /chat 改完上下文树
      await handleReply(ctx, {mention}) // ← 等待在它内部、buildContext 之前
```

`handleReply` 内（`checkIfMentioned` / `shouldSkipMessage` 之后、`createChatContext` 之前）：

```ts
// 此时 /chat 已改完 replies，树已定型
await awaitContextMedia(ctx);          // 收集整棵树 → 等待 + 反馈
const chatContext = await createChatContext(ctx);  // 发 processing 占位
```

- **pic 路径**等 `[当前, reply]`（它们的参考图只来自当前消息和被回复消息，且 `checkPic*` 会读图，必须先就绪）。
- **普通 / `/chat`** 等**整棵上下文树**（在 `dealChatCommand` 之后收集，自然含 `/chat` 引入的消息）。
- 各路径只等一次（pic 与普通/ chat 互斥）。

### 3.3 收集上下文树涉及的消息：`collectContextMessageIds`

复用 `getRepliesHistory`（它就是 `buildContext` 用的同一套遍历），再补上各消息 `replies` 里的 media-group 子图：

```ts
const collectContextMessageIds = async (chatId, messageId): Promise<number[]> => {
    const history = await getRepliesHistory(chatId, messageId, { excludeSelf: false });
    const ids = new Set<number>([messageId]);
    for (const m of history) {
        ids.add(m.messageId);
        try { (JSON.parse(m.replies) as number[]).forEach((r) => ids.add(r)); } catch {}
    }
    return [...ids];
};
```

> 代价：`getRepliesHistory` 会被多跑一次（这里 + `buildContext` 内）。上下文树通常只有几条~几十条消息，可接受；若要省，可让 `buildContext` 接收预取的 history（本期先不做）。

### 3.4 核心：`awaitMediaWithFeedback(ctx, ids)`

```ts
const FEEDBACK_TIMEOUT_MS = 30000;

const awaitMediaWithFeedback = async (ctx, ids: Array<number | undefined>) => {
    const pending = ids.filter((id): id is number => typeof id === 'number')
                       .filter((id) => getAsyncFileSaveMsgIdList().includes(id));
    if (pending.length === 0) return;                       // 没有在下载的 → 不提示

    const notice = await ctx.reply('⏳ 正在获取消息中的媒体，请稍候…', {
        reply_parameters: { message_id: ctx.message.message_id },
    });

    const start = Date.now();
    while (Date.now() - start < FEEDBACK_TIMEOUT_MS) {
        if (pending.every((id) => !getAsyncFileSaveMsgIdList().includes(id))) break;
        await new Promise((r) => setTimeout(r, 100));
    }

    const failures: string[] = [];
    for (const id of pending) {
        const stuck = getAsyncFileSaveMsgIdList().includes(id);
        const msg = await getMessage(ctx.chat.id, id);
        if (stuck || !msg?.fileUniqueId) failures.push(describeFailure(msg, stuck)); // [system] 标记 / 超时
    }

    if (failures.length === 0) {
        await ctx.api.deleteMessage(ctx.chat.id, notice.message_id).catch(() => {});   // 无缝
    } else {
        await ctx.api.editMessageText(ctx.chat.id, notice.message_id,
            `⚠️ 以下媒体获取失败，不会纳入本次对话上下文：\n${failures.map((f) => `• ${f}`).join('\n')}`
        ).catch(() => {});
    }
};
```

`describeFailure`：从消息文本的 `[system] …` 提取原因（超大/下载失败/渲染失败）；仍在队列记为「下载超时」。删除成功后 `createChatContext` 发的 processing 占位无缝接上；失败时提示保留为报错、占位另发其下。

## 4. 各路径行为

| 路径 | 等待时机 | 等待范围 | 反馈 |
|---|---|---|---|
| 普通 @bot / 回复 bot | `handleReply` 内、`buildContext` 前 | **整棵 reply 树**（`getRepliesHistory`）| ✅ |
| `/chat` | 同上（`dealChatCommand` 改树之后）| **整棵树（含 `/chat` 引入的消息）** | ✅ 正确覆盖改树 |
| `/picbanana`、`/picgpt` | `checkPic*` 读图之前 | 当前消息 + reply 目标（参考图） | ✅ |
| 未 @bot 的群消息 | — | — | 不触发（`willReply` 拦截）|

## 5. 需要你拍板的点

1. **超时阈值** `FEEDBACK_TIMEOUT_MS = 30000`（与单文件下载超时一致）是否合适？
2. **提示文案**：「⏳ 正在获取消息中的媒体，请稍候…」/ 失败「⚠️ 以下媒体获取失败，不会纳入本次对话上下文：…」是否调整。
3. **`getRepliesHistory` 多跑一次**是否接受（vs 改 `buildContext` 接收预取 history，改动更大）。

## 6. 影响与风险

| 改动 | 风险 | 缓解 |
|---|---|---|
| 幂等前移 + 移除 `handleReply` 内标记 | 若 `handleReply` 有其它调用方会失去幂等 | 已确认仅此一处调用；retry 走 callback 不经此路径 |
| 等待下沉 + 收集整棵树 | `getRepliesHistory` 多一次遍历 | 树规模小；必要时后续优化 |
| 发/删/编辑提示 | 删除/编辑可能失败（消息已删） | 均 `.catch(()=>{})` |
| pic 路径缺 prompt 时仍会等媒体 | 轻微浪费（少见） | 可接受；如需可在 prompt 校验后再等 |

## 7. 验证方式

1. 发大图+@bot：先「正在获取媒体」→ 下载完无缝变「Thinking」。
2. 发超大视频(>20MB)+@bot：提示就地变「⚠️…超大文件」，回答照常。
3. 纯文本@bot：无提示，直接 Thinking。
4. 群里别人发媒体没@bot：无任何提示。
5. **`/chat 5` 回复一条带图消息、且后续若干条也带刚发的图**：等待覆盖被回复消息**和** `/chat` 纳入的后续消息的媒体（验证改树后的范围正确）。
6. `/picbanana` 回复参考图：参考图就绪后再开始生成。
