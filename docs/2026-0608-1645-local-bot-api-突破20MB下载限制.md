# Local Bot API 突破 20MB 下载限制（2026-0608-1645）

## 1. 设计目标（从用户视角）

用户 @bot 发送或引用**最大 200MB** 的视频 / 文件 / 长语音 / 大贴纸时，bot 能像现在处理小文件一样正常获取并纳入对话，而不再因 Telegram 云端 `getFile` 的 **20MB 上限**直接「获取失败」。

这套能力**完全可选、优雅降级**：

- **配置了**自建 Local Bot API（填了 `api_id` / `api_hash` / `apiRoot`）→ 启用，下载上限 200MB、直接读盘。
- **没配置** → **无缝回退**到现在的云端 `getFile`（20MB、走 SOCKS 代理），行为与现状**一字不差**，零风险。

同时不影响同机的 luoxu-bot：复用它的 `api_id` / `api_hash` **不会导致掉线**（见 §4.2）。

## 2. 背景：20MB 从哪来

- **根因**：云端 `api.telegram.org` 的 `getFile` 下载上限写死 20MB，无法绕过。
- **当前链路**（`src/db/autoSave.ts`）：
  - `MAX_MEDIA_BYTES = 20 * 1024 * 1024`，`sizeBytes` 超过即 `too_large`，不下载。
  - `downloadTelegramFile`：`bot.api.getFile(fileId)` → 拼 `https://api.telegram.org/file/bot<token>/<file_path>` → `https.get` 走 `SocksProxyAgent`（`BOT_PROXY=socks5h://warp:1080`）。
- **唯一官方突破手段**：自建 [telegram-bot-api](https://github.com/tdlib/telegram-bot-api) 并以 `--local` 模式运行 → 下载上限提到 **2000MB（2GB）**，且 `getFile` 返回**服务器本地绝对路径**，直接读盘、连下载都省。

> 代理不是障碍：本机直连 Telegram 通（warp 只是网络优化），bot-api（TDLib）可直连 DC。

## 3. 方案总览：环境变量驱动的双模式

以单一开关 `TG_LOCAL_API_ROOT` 是否配置来切换；判定「读盘 vs 下载」用 `file_path` 是否为绝对路径（`--local` 一定返回绝对路径），天然兼容。

| | 云端模式（默认 / 未配置） | Local 模式（配置后） |
|---|---|---|
| 触发条件 | 未设 `TG_LOCAL_API_ROOT` | 设了 `TG_LOCAL_API_ROOT` |
| grammy `apiRoot` | 默认 `api.telegram.org` | `http://telegram-bot-api:8081` |
| bot 出网 | 走 `BOT_PROXY`（SOCKS） | bot→bot-api 走**内网**（无需 SOCKS）；bot-api→TG 直连 |
| `getFile` 返回 | 相对 `file_path` | 绝对本地路径 |
| 取字节方式 | `https` 下载（SOCKS） | `fs.readFile`（共享卷） |
| 大小上限 | 20MB | **200MB** |

## 4. 详细设计

### 4.1 新增容器 `telegram-bot-api`（`docker-compose.yaml`）

与 `tgs-converter` 同套路追加一个服务。**只有**配齐 `.env` 三项才会被启用（否则可不启动此容器，k-on-bot 自动走云端）。

```yaml
  telegram-bot-api:
    container_name: telegram-bot-api
    image: aiogram/telegram-bot-api:latest
    restart: always
    environment:
      - TELEGRAM_API_ID=${TG_API_ID}
      - TELEGRAM_API_HASH=${TG_API_HASH}
      - TELEGRAM_LOCAL=1          # 开启 --local：2GB 上限 + 返回本地绝对路径
      - TZ=Asia/Shanghai
    volumes:
      - tg-bot-api-data:/var/lib/telegram-bot-api   # 落盘的文件
    networks:
      - traefik-net
    # 可选：想让它也走 warp 优化（非必需，本机直连通）
    # network_mode: "service:warp"

  k-on-bot:
    # ...新增：
    environment:
      - TG_LOCAL_API_ROOT=http://telegram-bot-api:8081   # ← 启用 local 模式的开关
    volumes:
      - tg-bot-api-data:/var/lib/telegram-bot-api        # ← 同卷同路径，供读盘
    depends_on:
      - tgs-converter
      - telegram-bot-api

volumes:
  tg-bot-api-data:
```

**关键点**：两个容器把同一卷挂到**相同路径** `/var/lib/telegram-bot-api`，这样 `getFile` 返回的绝对路径在 k-on-bot 容器里直接可读。

### 4.2 凭据复用（为什么不冲突）

| | 登录主体 | session 存在哪 |
|---|---|---|
| luoxu-bot | 你的**用户账号**（Telethon user） | `luoxu-bot/mybot.session` |
| 本方案 bot-api server | **bot**（`BOT_TOKEN`） | `tg-bot-api-data` 卷内 |

- `api_id` / `api_hash` 标识「一个应用」，**设计上允许被任意多会话复用**，不是会话凭证、复用不掉线。
- 两者登录的是**不同主体**（user account vs bot），各自独立 session，互不接触。
- **做法**：把 `luoxu-bot/config.toml` 的 `[telegram]` 段 `api_id` / `api_hash` 抄进 **k-on-bot 的 `.env`**：
  ```
  TG_API_ID=<复用 luoxu 的，本文档不记录其值>
  TG_API_HASH=<同上>
  TG_LOCAL_API_ROOT=http://telegram-bot-api:8081
  ```
- 凭据只放 `.env`（已被 `env_file` 注入），**不写进 compose / 文档明文**。

### 4.3 bot 连接（`src/index.ts`）

按 `TG_LOCAL_API_ROOT` 切换 `apiRoot`；local 模式连的是内网，**不挂 SOCKS agent**。

```ts
const proxyUrl = process.env.BOT_PROXY;
const localApiRoot = process.env.TG_LOCAL_API_ROOT;

const botConfig = localApiRoot
  ? {
      // Local Bot API: talk to the in-cluster server over plain HTTP, no proxy.
      client: { apiRoot: localApiRoot },
    }
  : proxyUrl
    ? {
        // Cloud Bot API: keep the SOCKS proxy (undici fetch can't speak SOCKS).
        client: { baseFetchConfig: { agent: new SocksProxyAgent(proxyUrl), compress: true } },
      }
    : undefined;

const bot = new Bot(process.env.BOT_TOKEN, botConfig);
```

### 4.4 下载链路（`src/db/autoSave.ts`）

**(a) 大小上限按模式取值**：

```ts
// Cloud getFile caps downloads at 20MB; a self-hosted --local server raises it.
const MAX_MEDIA_BYTES = process.env.TG_LOCAL_API_ROOT
    ? 200 * 1024 * 1024
    : 20 * 1024 * 1024;
```

**(b) `downloadTelegramFile` 双模式**——`file_path` 绝对路径→读盘；否则保留原 `https` 下载：

```ts
import { readFile } from 'node:fs/promises';

const localApiRoot = process.env.TG_LOCAL_API_ROOT;
const fileBaseUrl = localApiRoot
    ? `${localApiRoot}/file/bot${process.env.BOT_TOKEN}`
    : `https://api.telegram.org/file/bot${process.env.BOT_TOKEN}`;
// SOCKS agent only used for the cloud HTTPS path.
const downloadProxyAgent = (!localApiRoot && process.env.BOT_PROXY)
    ? new SocksProxyAgent(process.env.BOT_PROXY)
    : undefined;

const downloadTelegramFile = async (bot: Bot, fileId: string): Promise<Buffer> => {
    const file = await bot.api.getFile(fileId);
    const filePath = file.file_path;
    if (!filePath) throw new Error('getFile returned no file_path');

    // --local server returns an absolute on-disk path → read directly from the
    // volume shared with the bot-api container (no network round-trip).
    if (filePath.startsWith('/')) {
        return readFile(filePath);
    }

    // Otherwise download over HTTP(S). Cloud → api.telegram.org via SOCKS;
    // a local server that somehow returned a relative path → its /file endpoint.
    const url = `${fileBaseUrl}/${filePath}`;
    return new Promise<Buffer>((resolve, reject) => {
        const request = https.get(url, { agent: downloadProxyAgent, timeout: DOWNLOAD_TIMEOUT_MS }, (res) => {
            if (res.statusCode !== 200) { res.resume(); reject(new Error(`file download HTTP ${res.statusCode}`)); return; }
            readStreamToBuffer(res).then(resolve, reject);
        });
        request.on('timeout', () => request.destroy(new Error('file download timed out')));
        request.on('error', reject);
    });
};
```

> 注：local 模式下 `fileBaseUrl` 用的是 `http://`，而上面示例的兜底分支写死了 `https.get`。实现时这个兜底分支按 `localApiRoot` 选 `http`/`https` 模块（local 内网用 `http`）。绝对路径读盘是主路径，兜底极少触发。

### 4.5 一次性迁移（`logOut`）

同一个 `BOT_TOKEN` **不能同时**在云端和本地两个 server 跑 `getUpdates`。切换步骤（官方语义）：

1. **停**当前 k-on-bot（停掉云端 `getUpdates`）。
2. 对**云端**调一次 `logOut`：`https://api.telegram.org/bot<token>/logOut`（走 warp 代理）。
   - 成功后可立即登录 local server；但 **10 分钟内不能再登回云端**。
3. 配好 `.env` 三项 + 启动 `telegram-bot-api` 容器，再启 k-on-bot（此时连 local server）。

**回退到云端**：删掉 `.env` 的 `TG_LOCAL_API_ROOT` 重启即可让代码走云端分支；但因 `BOT_TOKEN` 之前已从云端 `logOut`，需对 local server 先 `logOut`/`close`、再让云端重新接管（**此回退路径需实测确认**，文档先标注）。

### 4.6 磁盘清理

`--local` 模式下载的文件**落盘后不自动删除**。本项目已有 `MediaCache`（字节进 SQLite），所以：

- **推荐**：autoSave 把字节写入 `MediaCache` 成功后，**立即 `unlink` 掉 `file_path` 对应的盘文件**（一次性消费）。
- 备选：cron 定期清 `tg-bot-api-data` 里 N 天前的文件。

## 5. 下游约束（重要，须先评估）

**突破 `getFile` ≠ 模型吃得下 200MB**：

- 多数多模态模型对 **inline data** 有上限（如 Gemini inline 约 20MB，超过须改用 File API 上传）。
- 大文件 base64 进 `MediaCache`（SQLite BLOB）+ 进 context，会显著拉高内存 / token / 成本，`database.sqlite` 已 ~922MB。
- 因此 **200MB 是「下载层」上限**；「喂模型层」可能仍受模型 inline 限制。

**范围建议**：本期只做「下载层」（让大文件能下载、缓存、用于非 inline 用途或小于模型上限的部分）；模型 **File API** 接入改动更大，建议**后续单独立项**，不混进本期。

## 6. 各场景行为

| 场景 | 未配 local（现状） | 配了 local |
|---|---|---|
| 发 10MB 图 | ✅ 下载 | ✅ 读盘 |
| 发 30MB 视频 | ❌ `too_large` 提示 | ✅ 读盘纳入 |
| 发 150MB 文件 | ❌ `too_large` | ✅ 读盘纳入（受 §5 模型约束）|
| 发 >200MB | ❌ `too_large` | ❌ `too_large` |
| luoxu-bot | 不受影响 | **不受影响**（不掉线）|

## 7. 验证方式

1. **未配 local**：发 30MB 视频 → 仍 `too_large`，行为等价现状（证明 fallback 无损）。
2. **配 local**：发 ~100MB 视频 + @bot → 「正在获取媒体」→ 读盘成功 → 正常回答。
3. luoxu-bot 在迁移前后均不掉线、搜索正常。
4. 发 >200MB → `too_large` 提示。
5. 确认盘文件在写入 `MediaCache` 后被清理（磁盘不持续上涨）。

## 8. 风险与回滚

| 项 | 风险 | 缓解 |
|---|---|---|
| 未配 local | 无（等价现状） | 默认即现状，零风险 |
| 共享卷路径不一致 | 绝对路径读不到 | 两容器同路径挂载；读盘失败兜底走 `http://bot-api/file` |
| `logOut` 迁移 | 不可逆敏感操作（从云端登出）、10 分钟锁 | 低峰期执行；回退路径先实测 |
| 磁盘上涨 | local 文件不自删 | §4.6 消费即删 |
| 下游模型 inline 限制 | 大文件下载成功但喂模型失败 | §5 限定本期范围；超限文件的模型侧后续立项 |

## 9. 需你拍板的点

1. **范围**：本期是否只做「下载层 200MB」，模型 File API 后续单独立项？（建议：是）
2. **bot-api 出网**：直连 TG，还是 `network_mode: "service:warp"` 走 warp 优化？（建议：先直连，简单）
3. **磁盘清理**：写入 `MediaCache` 后即删盘文件（推荐）vs cron 定期清？
4. **迁移窗口**：`logOut` 会让 bot 从云端登出且 10 分钟不能回退，确认在低峰期执行。
