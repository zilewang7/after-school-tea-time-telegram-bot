# 借 luoxu MTProto 取数层：链接预览（含媒体）注入 AI 上下文 — 设计文档

> 2026-0609-1745
> v2：存储改为落 k-on db，对齐现有媒体体系
> v3：链接预览按 URL 缓存（同链接只取一次）；bot 消息走 Bot API 9.x 新能力 + 一致逻辑

## 1. 设计目标（end-user 视角）

群里有人发带链接的消息时，k-on-bot 现在只能看到一条光秃秃的 URL，看不到 Telegram 已经替我们抓好的网页预览（标题、描述，以及预览自带的图片/视频）。结果模型只能靠链接以外的上下文瞎猜，回复经常脱离链接内容、答非所问。

本方案让 k-on-bot 借助**同机的 luoxu**（它以 MTProto user 账号运行，能拿到完整预览），取到链接预览的**全部内容（文字 + 媒体）**，存入自己的库、纳入上下文喂给模型，让回复真正“看过”这个链接。同链接只取一次（按 URL 缓存）。未接入 luoxu 时，行为与现状完全一致（优雅降级，零影响）。

---

## 2. 范围

**做：**

- 链接预览（web preview）的**全部内容**：文字（`url/site_name/title/description/type`）+ 媒体（`webpage.photo`/`webpage.document`）。
- **luoxu（provider）不落库**：现场 `get_messages` + `download_media`、流式返回、用完即弃。
- **k-on（消费方）落库 + 按 URL 缓存**：预览存进 `LinkPreviewCache`（URL 作主键），媒体进 MediaCache（大文件 GCS）；**同一链接只取一次**，后续命中缓存直接用、不再调 luoxu。复用现有清理体系。
- **bot 消息**：借 Bot API 9.x 新能力收取群内其它 bot 的消息，与用户消息**走一致逻辑**（autoSave → 含链接取预览）；**不**让 luoxu 反向写 k-on 库。

**不做 / 不变：**

- 用户/bot **直接发**的普通媒体（图/视频/文件）：维持 k-on 现有链路（Bot API getFile + local bot api + GCS）。
- 不改 luoxu 的采集/索引逻辑、不让 luoxu 为本功能持久化任何东西。

---

## 3. 关键事实与约束（已读双方源码 + 查证）

- Bot API 不返回 webpage 解析内容（仅 URL 字符串 + `link_preview_options` 开关）；预览内容只在 MTProto 层。
- luoxu = telethon（MTProto user `+44…`），`msg.web_preview` 可拿完整 `WebPage`（含 `photo`/`document`）。
- luoxu 的 `api` 容器 = `python -m luoxu`（`Indexer.run`），**同进程持有 MTProto client + 跑 9008 aiohttp** → 加端点、用现成 client 现拉，可行，复用唯一 `mybot.session`（session 不能并发 → 取数层只能寄生此进程）。
- 群 id 转换：luoxu 用裸 `channel_id`，k-on chatId 为 `-100`+`channel_id`。
- **Bot 看其它 bot 消息（已查证）**：Bot API 9.x（2026）新增“可见群内其它 bot 的 **certain** 消息”（changelog；同批 `getChatAdministrators` 加 `return_bots`）。此前是硬限制（admin/privacy-off 也收不到）。
  - **caveat**：是 “certain messages”、非全量；确切条件（是否需 admin/关 privacy、覆盖哪些 bot 消息）**待开启后实测**。不阻塞 k-on 实现（autoSave 收到即处理）。
- **群范围约束**：只有 luoxu user 账号在的群能现拉预览；其它群 luoxu 返回取不到 → k-on 降级。

## 4. 数据流（URL 缓存优先 + 融入现有 autoSave / build / autoClear）

**入库（autoSave，含链接才触发；用户与 bot 消息一致）：**
```
收到消息（含其它 bot 的）
  ├─ 普通媒体（现状）→ acquireMediaBytes → MediaCache
  └─ 含链接（新增）→ 异步 acquireLinkPreview：
        提取首个 URL
        ├─ 查 LinkPreviewCache[url] 命中 → 结束（不调 luoxu）
        └─ 未命中 → addAsyncFileSaveMsgId(msgId)
              luoxu GET /preview?g=&id=  （现拉 web_preview；ready/pending/none）
              ├─ pending → 轮询（§5：等到确认，无上限）
              ├─ none    → 写一条“无预览”缓存（避免反复重试）
              └─ ready   → 取文字；有媒体则 GET /preview-media → inline/GCS → MediaCache(key=urlHash)
                         → upsert LinkPreviewCache[url] = {title,description,siteName,type,mediaKey?,...}
              removeAsyncFileSaveMsgId(msgId)
```

**取用（context-builder，从缓存读）：**
```
提取消息 text 的 URL → 查 LinkPreviewCache[url]：
  ├─ 文字 → 注入 [system] 链接预览 text part（正文之后）
  └─ mediaKey → MediaCache 读 → image/media part；缺失（媒体已过期）则只注入文字
```

**清理（autoClear，复用）：** 预览媒体在 MediaCache，享受“媒体/GCS 1 天、LRU 7 天”；`LinkPreviewCache` 行按 `lastUsedAt` LRU 清理（与 MediaCache 同节奏）。

---

## 5. 时序：等到确认（无超时上限）

- **仅“含链接”消息触发**；无链接零延迟。**缓存命中时无需取数/等待**。
- **状态驱动**：webpage 三态 `WebPage`(ready)/`WebPagePending`(pending)/`WebPageEmpty`(none)，luoxu 端点映射返回。
- **等到确认（无固定超时）**：仅 `pending` 时轮询（间隔 1s——本地无开销），直到终态。
  - `pending` 是临时态，必然转 ready 或 none，自然终止。
  - **提前退出**：取数请求失败（消息不存在/群不可达/luoxu 报错）→ 立即降级。
- 复用 `addAsyncFileSaveMsgId`/`waitForFileSave` 让紧跟回复等预览就绪；保留高额 backstop 防 luoxu 无响应时永久阻塞。

## 6. luoxu 端改动（本地源码，可改、可访问 client）

- 取数函数 `fetch_preview(client, channel_id, msgid)`：`get_messages(channel_id, ids=[msgid])` → 读 `msg.web_preview` → 按 `WebPage/WebPagePending/WebPageEmpty` 返回 `{status, url, site_name, title, description, type, media:{kind,mime,size}|None}`。
- `web.py`：
  - `PreviewHandler`：`GET {prefix}/preview?g=&id=` → 上述 JSON。
  - `PreviewMediaHandler`：`GET {prefix}/preview-media?g=&id=&which=photo|doc` → `download_media(..., file=BytesIO())` 流式字节、不写盘。
  - `setup_app` 注册路由，复用 indexer 进程的 `client`。
- 网络：见 §8。

## 7. k-on-bot 端改动

- **env**：`LUOXU_PREVIEW_URL`（空 → 关闭）；补 `env.d.ts`、`.env.example`、`docker-compose.yaml`。
- **新表 `LinkPreviewCache`**（content-addressed by URL）：
  - `url`（PK）、`title`、`description`、`siteName`、`type`、`mediaKey`(→MediaCache)、`mediaMime`、`mediaKind`、`status`('ready'|'none')、`createdAt`、`lastUsedAt`。
  - `sync({alter:true})` 自动建表。**Message schema 不动**。
- **`luoxu-preview-service.ts`**：`fetchLinkPreview(chatId, messageId)`：提取首个 URL → 查 `LinkPreviewCache` 命中即返回 → 否则 id 转换 + 调 `/preview`（pending 轮询）→ 有媒体则 `/preview-media` 取字节 → `size>INLINE_MAX_BYTES ? uploadBytesToGcs : putCachedMedia(inline)`（key=`luoxu-preview:{sha256(url)}:{which}`）→ upsert `LinkPreviewCache` → 返回；失败返回 `null`。
- **autoSave**：在 media 异步分支旁加“含链接”异步分支，调 `fetchLinkPreview`；接入 `addAsyncFileSaveMsgId`/`removeAsyncFileSaveMsgId`。
- **bot 消息**：核对 autoSave 中间件不把其它 bot 消息误判 `fromBotSelf`、不被 `excludeList`/过滤排除（仅排除 k-on 自己）；它们作为普通消息进上下文并同样触发链接预览。
- **context-builder**：`buildMessageContent`/`buildCurrentMessageContent` 提取 URL → 查 `LinkPreviewCache` → 注入文字 part（正文之后）+ `mediaKey` 经 MediaCache 解析的 image/media part（音视频复用 `AUDIO_VISUAL_NUDGE`）。
- 全程 `try/catch` + 失败降级。

luoxu `api` 服务加入共享 external 网络（`traefik-net`），设 `container_name: luoxu-api`；k-on 配 `LUOXU_PREVIEW_URL=http://luoxu-api:9008/luoxu`。

## 9. 优雅降级矩阵

| 情况 | 行为 |
|---|---|
| 未配 `LUOXU_PREVIEW_URL` | 跳过，**完全等于现状** |
| 消息无链接 | 不触发，零延迟 |
| URL 命中缓存 | 直接用，不调 luoxu |
| 群不在 luoxu / 取不到 | 降级，不存预览 |
| webpage = none | 写 'none' 缓存，不注入 |
| luoxu 不可达 / 报错 | 降级，不注入 |
| 预览媒体下载失败 | 只存文字、跳过媒体 |
| 预览媒体 GCS 1 天后被清 | build 时降级为只注入文字 |
| Bot API 收不到某 bot 消息（超出 “certain” 范围） | 该消息不进上下文（与现状一致） |

## 12. 预览媒体与视频获取边界（v4 补充）

- **预览媒体统一走 luoxu `download_media`**：包括 fixupx / vxtwitter 这类“Telegram 已内嵌、可直接播放”的视频——它们的本体已被 Telegram 抓存为 `webpage.document`，`download_media` 直接可得。无需 yt-dlp。
- **不引入 yt-dlp**：纯外链 iframe embed（YouTube/Vimeo 等，本体在源站）只给模型 `embed_url` + 缩略图 + 标题/描述，不下视频本体。
- **Instant View 全文**：`cached_page` 存在时，全量抓正文（blocks 拉平为文本，**不设字符上限**，token 不限）+ 顺带提取其中的媒体（photos/documents）；`cached_page` 是否随 `get_messages` 直接可得，实现时**实测**，拿不到则降级摘要。
- **媒体 which 标识**：`photo`/`doc`（webpage 主媒体）、`page_photo_{i}`/`page_doc_{i}`（cached_page 内媒体）；k-on 按 which 调 `/preview-media` 取字节，再走 inline/GCS。

## 13. 隐藏链接与原始文本还原（v5 补充）

- **问题**：text_link 实体（超文本链接）的 URL 不在消息文本里，存库后 build 时提不出 URL → 预览无法注入。
- **方案**：autoSave 存库前把 text_link 还原为内联 markdown `[标签](url)`（编辑消息的 autoUpdate 同样处理）。依据：其它格式实体（加粗/斜体/代码等）不丢字符，文本即原文；唯独 text_link 丢 URL——还原它即恢复完整原始信息。
- **收益**：存库/喂模型/URL 提取共用同一份完整文本；预览触发统一从 baseText（即将存库的字符串）提取，保证存取两端提取结果一致；模型即使无预览也能直接看到 URL。
- **边界**：本改动只影响新消息；历史消息存的是无链接文本，不回填。
